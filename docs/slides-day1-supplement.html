Policy Inheritance

Enterprise → Organization → Repository cascade

┌─────────────────────────────┐ │ ENTERPRISE (top-level) │ Sets the floor │ Enforce · No Policy · Disable│ └──────────────┬──────────────┘ │ inherits ▼ ┌──────────────┴──────────────┐ │ ORGANIZATION │ Can add restrictions, │ Enabled · Disabled · Unconfigured │ never relax └──────────────┬──────────────┘ │ inherits ▼ ┌──────────────┴──────────────┐ │ REPOSITORY │ Content exclusions, │ (+ Individual prefs) │ personal settings └─────────────────────────────┘
  • Enforced — mandatory baseline; orgs cannot change
  • No Policy / Allowed — delegated; org owners decide
  • Disabled — blocked enterprise-wide; orgs cannot enable
Key principle: child entities can only ADD restrictions, never relax ones from above. Absence of a restrictive policy ≠ explicit permission. Enforced policies create a "compliance floor" that prevents policy drift. The decision tree short-circuits: if enterprise denies, org/repo are never evaluated.

Copilot Organization Policies

Key levers for org owners (subject to enterprise constraints)

Feature Policies

  • Access policy — who gets Copilot: all members or selected
  • Code completions — IDE inline suggestions
  • Copilot Chat — IDE, GitHub.com, Mobile, CLI
  • Code review — AI-assisted PR reviews

Security-Sensitive Policies

  • Public code matching — block suggestions matching public code
  • Cloud agent — autonomous code changes (evaluate carefully)
  • Agent mode (IDE) — multi-step agentic workflows
  • MCP servers — external integrations (security review first)
⚠️ Security-by-default: Block public code matching, disable cloud agent & MCP servers until reviewed. Delegate agent mode to org owners.
Each policy has 3 states at org level: Enabled, Disabled, Unconfigured. Unconfigured = disabled until explicitly set. Best practice: explicitly configure ALL policies rather than leaving them unconfigured. Privacy policies: block prompt/suggestion collection to maintain data privacy (code is not retained). Preview features should be disabled in production.

Content Exclusions

Prevent Copilot from accessing sensitive files

How It Works

  • Glob patterns exclude files from AI context
  • Inline suggestions disabled for excluded files
  • Chat cannot reference excluded content
  • Configurable at repo, org, and enterprise level

Common Patterns

  • **/.env, **/.env.*
  • **/secrets/**, **/*credential*
  • **/*.pem, **/*.key
  • **/terraform.tfvars, **/*.tfstate

Key Details

  • Propagation delay: ~30 minutes (or reload IDE)
  • Enterprise exclusions apply to all users regardless of licensing org
  • Org exclusions apply only to users licensed through that org
⚠️ Exclusions do NOT apply to: Copilot CLI, coding agent, or agent mode. Disable those features if exclusion is critical for compliance.
After configuring exclusions, wait ~30 min or reload the IDE. To test: open an excluded file — no inline suggestions should appear, and Copilot Chat should not be able to use the file content. Enterprise-level exclusions are the strongest: they apply to ALL Copilot users in the enterprise. The gap with CLI/agent mode/coding agent is a critical security consideration — call it out explicitly.

Seat Management

Assignment Methods

  • All members — automatic for everyone in org
  • Selected members/teams — granular control
  • Direct enterprise assignment — for non-GHE licensed users
  • Self-service claiming — team join or issue-based requests

Cost Optimization

  • Reclaim after 30+ days inactivity
  • Set seat caps to limit consumption
  • Pilot programs before full rollout

Monitoring & APIs

  • Enterprise Insights tab — active users, feature adoption
  • Billing API: GET /enterprises/{ent}/copilot/metrics/...
  • 28-day usage reports per user and enterprise

License Inheritance

  • Access through org membership
  • Multiple org memberships → most permissive policy wins
  • Enterprise policies can override org-level controls
Key cost tip: identify users who haven't used Copilot recently, send reminders, then reclaim. The billing API provides 28-day rolling reports. Team-based assignment is the most scalable approach — team membership changes automatically adjust Copilot access. Self-service via GitHub App or issue templates reduces admin overhead and accelerates adoption.

Audit Log Overview

What's Captured

  • Settings changes & access control
  • User membership & role changes
  • Repository management events
  • App permissions & webhooks
  • Git events (push, clone, fetch)
  • Authentication & billing events

Retention & Access

TypeRetentionAccess
Web events180 daysUI, API, streaming, export
Git events7 daysAPI only, streaming
StreamedIndefiniteYour external storage
⚠️ Git events have only 7-day retention — always enable streaming for long-term capture.
Enterprise audit log aggregates across ALL child orgs — it's the primary tool for enterprise-wide monitoring. Org audit log is more focused for org admins without enterprise access. Default UI shows 3 months; use the `created` qualifier to access up to 180 days. Enterprise owners must explicitly enable IP address disclosure — it's off by default. EMU enterprises get additional user security log events in the enterprise log.

Searching the Audit Log

Phrase qualifiers for precise filtering

Key Qualifiers

QualifierExample
action:action:repo.create
actor:actor:octocat
created:created:>=2025-01-01
repo:repo:my-org/my-repo
operation:operation:authentication
country:country:US

Example Queries

  • Repo creation by user in last 30 days:
    action:repo.create actor:octocat created:>=2025-05-01
  • Branch protection changes:
    action:protected_branch.update created:>=2025-04-01
  • Member removals:
    action:org.remove_member created:>=2025-01-01
Multiple qualifiers are AND'd together. Date ranges: created:2025-01-01..2025-03-31
Qualifiers are space-separated and AND'd. The `created` qualifier is crucial — default UI only shows 3 months, but events exist for 180 days. Export options: JSON or CSV from UI for web events, compressed JSON for Git events. API rate limit: 1,750 queries/hr per user+IP. GraphQL supports org-level audit log queries with ~90-120 days of data.

Audit Log Streaming

Real-time export to external SIEM / storage for compliance

Supported Targets

ProviderAuth
Amazon S3Access keys or OIDC
Azure Blob StorageSAS token
Azure Event HubsConnection string
SplunkHEC token
Google Cloud StorageService account JSON
DatadogAPI key
HTTPS EndpointHEC token

Why Streaming Matters

  • Overcomes 180-day / 7-day retention limits
  • Required for SOC 2, HIPAA, PCI-DSS compliance
  • Enables real-time alerting in SIEM tools
  • Supports up to 2 endpoints simultaneously

Buffering Rules

  • Paused < 7 days → no data loss
  • Paused 7–21 days → resumes from 1 week back
  • Paused > 3 weeks → resets to current time
  • Datadog: max 18-hour pause window
Stream format: compressed JSON (.json.gz), at-least-once delivery, deduplicate via `_document_id`. Health checks run every 24 hours — you get 6 days to fix a misconfigured stream before events are dropped. Multi-endpoint (preview): stream to S3 for storage AND Splunk for alerting simultaneously. Azure Event Hubs is ideal for Microsoft Sentinel integration. Not supported in Azure Government (Blob + Event Hubs).

Dormant Users

30 days without qualifying activity = dormant

Qualifying Activities ✅

  • SAML SSO authentication
  • Creating/deleting repositories
  • Push to internal repo via HTTPS
  • Creating/closing issues & PRs
  • PR review comments
  • Starring a repository
  • Joining an organization

NOT Qualifying ❌

  • PAT-based access (CI/CD pipelines!)
  • SSH key-based Git operations
  • GitHub App-based workflows
  • Git ops on private repos
  • Reading/browsing without interaction
  • Public repo activity outside enterprise
⚠️ #1 gotcha: devs using PAT/SSH daily may appear dormant. Cross-reference audit log before reclaiming!
The 30-day window is fixed — cannot be configured. You cannot manually mark a user as active; they must perform a qualifying activity. Download the dormant users report from Enterprise → Compliance → Reports. Also viewable in Enterprise → People → filter "Dormant". License reclamation workflow: 1) Identify via report, 2) Cross-reference with audit log for PAT/SSH activity, 3) Confirm with managers, 4) Remove via UI or `removeEnterpriseMember` GraphQL mutation, 5) Clean up IdP access. For EMU: soft-deprovision via IdP (reversible) to free the license.
